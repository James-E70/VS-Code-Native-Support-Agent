The cause is confirmed: this is a Windows path length issue on the local workstations.

The failing file's name is 235 characters long:

Booking confirmation MSC COTONOU 925635 Lot 3 - S156361 --AW Master Packing List Lot 3 - 2B4-2026-022227 - 10730-OFS-0207 + 0208+ 0209- -SHORTLISTED 107302631-CONT OFF SHORE JAFURAH - GCP III - PO 1073005050 Air Compressor Packages.eml

When CargoWise opens an email document from eDocs in a WiseCloud environment, it transfers the file to the local workstation via the Remote Desktop Services file redirect channel and writes it to the local Windows temp folder using the email's subject line as the filename. The full path is constructed as:

C:\Users\[local username]\AppData\Local\Temp\[filename].eml

Adding the local temp path prefix (approximately 44 characters) to the 235-character filename results in a total path length of approximately 279 characters. Windows enforces a hard limit of 260 characters for fully qualified file paths. When this limit is exceeded, the temporary file creation fails silently — no log entry is generated because the failure occurs before the file-open handoff to Outlook. This is why the WiseCloud log shows a successful connection but no file-open event.

The 13-APR-26 file opens correctly because that email has a shorter subject line, keeping the total local temp path within the 260-character limit. The file size of 1.03 MB is not a factor.

Immediate workaround (already working for you):

When you right-click and save the file from eDocs, you choose the destination path on your local machine. Saving to a short path such as C:\Temp\ keeps the total path well within the 260-character limit and allows Outlook to open it normally. This will continue to work reliably for any affected email.

IT resolution (client-side fix):

On Windows 10 version 1607 or later, your IT team can remove the 260-character path limit on the affected workstations by enabling the following Group Policy:

Computer Configuration > Administrative Templates > System > Filesystem > Enable Win32 long paths > Enabled

After applying this policy and restarting the workstations, the local temp path length will no longer block file opens regardless of subject line length. Because this affects multiple users, deploying the policy via your domain Group Policy Objects would apply the fix to all affected workstations at once.
