When CargoWise opens an email document from eDocs in a WiseCloud environment, it writes a temporary file on the local workstation via the Remote Desktop Services file redirect channel using the email's subject line as the filename. The full Windows temp path is constructed as:

C:\Users\[username]\AppData\Local\Temp\[filename].eml

Adding the standard temp path prefix of approximately 50 characters to the 235-character filename results in a total path length of approximately 285 characters. Windows enforces a hard limit of 260 characters for fully qualified file paths. When this limit is exceeded, the file creation fails silently — no log entry is generated because the failure occurs before the file-open handoff to Outlook. This is why the WiseCloud log shows a successful connection but no file-open event.

The 13-APR-26 file opens correctly because that email has a shorter subject line, keeping the total temp path within the 260-character limit.

The file size of 1.03 MB is not a factor in this issue.

Immediate workaround (already working for you):

When you right-click and save the file from eDocs, you control where it is saved. Saving to a short path such as C:\Temp\ keeps the total path well within the 260-character limit and allows Outlook to open it normally. This will continue to work reliably as a workaround for any affected email.

IT resolution options:

1. Enable Windows Long Path support via Group Policy (recommended first step):
   On Windows 10 version 1607 or later, your IT team can remove the 260-character path limit by enabling the following Group Policy on the affected workstations:
   Computer Configuration > Administrative Templates > System > Filesystem > Enable Win32 long paths → Enabled
   After applying this policy and restarting the workstations, long temp paths will no longer cause this failure. Please confirm with your IT team whether this policy can be deployed.
